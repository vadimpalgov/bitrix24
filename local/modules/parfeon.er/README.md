## Типы заявок

Тип хранится в поле `UF_CRM_10_TYPE` — привязка к элементу инфоблока-справочника.

| Тип | Особенности |
|---|---|
| **Заявка на отпуск** | Проверка минимальной длительности и срока подачи. При одобрении создаётся запись в графике отсутствий. |
| **Заявка на отсутствие** | Без специфических проверок дат. |

## Смарт-процессы

| Сущность | Код настройки | Entity Type ID | Описание |
|---|---|---|---|
| **EmployeeRequest (ER)** | `ER_SMART_PROCESS_ID` | 1046 | Заявка сотрудника — основная сущность |
| **Approvers (AP)** | `AP_SMART_PROCESS_ID` | 1050 | Задача согласующего, дочерняя к ER |

## Жизненный цикл заявки

```
1.  Сотрудник создаёт ER-элемент
            ↓
2.  BEFORE_SAVE: CreateName
    → заголовок «{ТипЗаявки} от {Имя Фамилия}»
            ↓
3.  BEFORE_SAVE: DayCountChecker / DayStartChecker
    → проверки дат (только если LA_ENABLE_MIN_DAYS_CHECK = Y)
            ↓
4.  BEFORE_SAVE: DepartmentManagerResolver
    → цепочка руководителей по иерархии → HEAD_OF_DEPARTMENT
            ↓
5.  BEFORE_SAVE: HRManagersResolver
    → HR-менеджеры из подразделения «HR» → HR_MANAGERS
            ↓
6.  Сотрудник переводит заявку в стадию «В работе» (ER_START_STATUS)
            ↓
7.  AFTER_SAVE: ApproverResolver → CreateApprovers.createPhase(1)
    → создаются AP-элементы только для ФАЗЫ 1 (HR)
            ↓
8.  AFTER_SAVE (AP): SendNotify
    → IM-уведомление каждому согласующему фазы 1
            ↓
9.  Согласующий переводит свой AP в APPROVE или REJECT
            ↓
10. BEFORE_SAVE (AP): Approvers\ChangeStatus
    → управляет переходами между фазами и итоговым статусом ER
            ↓
11. После одобрения последнего согласующего финальной фазы:
    ER → ER_APPROVE_STATUS
            ↓
12. BEFORE_SAVE (ER): EmployeeRequest\ChangeStatus
    → IM-уведомление сотруднику об итоге
            ↓
13. AFTER_SAVE (ER): AddingAbsencesSchedule
    → запись в инфоблок «absence» (только APPROVE, только отпуск)
```

> При **любом REJECT** на любой фазе → ER немедленно переходит в `ER_REJECT_STATUS`.

## Три фазы согласования

Фазы выполняются строго последовательно. Следующая фаза создаётся только после завершения предыдущей. Номер фазы хранится в поле `UF_CRM_11_PHASE` каждого AP-элемента.

### Фаза 1 — HR

- AP-элементы создаются для **всех** пользователей из `HR_MANAGERS`.
- Все работают **параллельно**.
- Условие перехода к фазе 2: **хотя бы один** HR одобрил.

```
HR 1 ──╮
HR 2 ──┼──→ ≥1 APPROVE → создаётся Фаза 2
HR N ──╯
```

### Фаза 2 — Руководители проекта (РП)

- AP-элементы создаются для пользователей из поля `PROJECT_MANAGERS` (заполняется вручную в заявке).
- Все работают **параллельно**.
- Условие перехода к фазе 3: **все** РП одобрили.
- Если `PROJECT_MANAGERS` пустое — фаза **пропускается**, переход сразу к фазе 3.

```
РП 1 ──╮
РП 2 ──┼──→ все APPROVE → создаётся Фаза 3
РП N ──╯
```

### Фаза 3 — Руководители (последовательно)

- AP-элементы создаются **по одному**, снизу вверх по иерархии.
- Следующий согласующий получает AP только после того, как предыдущий одобрил.
- Порядок задаётся полем `UF_CRM_11_ORDER`.
- Условие завершения: последний в цепочке одобрил → ER переходит в `ER_APPROVE_STATUS`.

```
Руководитель 1 (ORDER=1) → APPROVE
        ↓ создаётся следующий
Руководитель 2 (ORDER=2) → APPROVE
        ↓
      ...
        ↓ создаётся следующий
Директор (последний)     → APPROVE → ER_APPROVE_STATUS
```

#### Источник списка в фазе 3

```
Есть UF_APPROVAL_PROFILE на заявителе?
         ↙                        ↘
        НЕТ                       ДА
         ↓                         ↓
   Иерархия отделов           Дочерние ALP-элементы
   (HEAD_OF_DEPARTMENT)        профиля PP,
   + настройки директора       отсортированные по UF_CRM_12_ORDER
```

#### Настройки директора (только при использовании иерархии)

| Условие | Поведение |
|---|---|
| `LA_EXCLUDE_DIRECTOR = N` | Директор включается в цепочку последним |
| `LA_EXCLUDE_DIRECTOR = Y` | Директор исключается |
| `LA_EXCLUDE_DIRECTOR = Y` + `LA_FORCE_DIRECTOR_IF_MANAGER = Y` + заявитель сам руководитель | Директор добавляется последним несмотря на исключение |

## Профили согласования

Профиль — ручной шаблон цепочки согласования. Если назначен пользователю, **полностью заменяет** автоматическую иерархию в фазе 3.

### Структура

**ProfileParticipants (PP)** — именованный шаблон:
- `TITLE` — название (например, «Согласование для топ-менеджеров»)
- Назначается сотруднику через поле `UF_APPROVAL_PROFILE` на сущности «Пользователь»

**ApprovalProfile (ALP)** — один шаг внутри шаблона (дочерний к PP):
- `UF_CRM_12_USER` — согласующий
- `UF_CRM_12_STAGE` — стадия AP, которую он должен выставить
- `UF_CRM_12_ORDER` — очерёдность в цепочке

### Пример

```
ProfileParticipants: «Согласование директора»
    ├─ ALP  ORDER=1  USER=Иванов (зам. директора)   STAGE=AP_APPROVE
    └─ ALP  ORDER=2  USER=Петров (генеральный)       STAGE=AP_APPROVE
```

При запуске фазы 3 сначала создаётся AP для Иванова. После его одобрения — AP для Петрова. После одобрения Петрова — ER переходит в APPROVE.

### Назначение профиля

На сущности «Пользователь» создаётся поле:

| Поле | Тип | Описание |
|---|---|---|
| `UF_APPROVAL_PROFILE` | Привязка к PP | ID профиля. Если пусто — используется иерархия. |

## Настройки модуля

Страница: `/bitrix/admin/settings.php?mid=parfeon.er`

### Вкладка «Настройки согласования»

| Ключ | Описание |
|---|---|
| `ER_SMART_PROCESS_ID` | ID смарт-процесса заявок (ER) |
| `ER_START_STATUS` | Стадия «В работе» — триггер запуска фазы 1 |
| `ER_APPROVE_STATUS` | Стадия одобрения ER |
| `ER_REJECT_STATUS` | Стадия отклонения ER |
| `AP_SMART_PROCESS_ID` | ID смарт-процесса согласующих (AP) |
| `AP_START_STATUS` | Начальная стадия AP |
| `AP_APPROVE_STATUS` | Стадия одобрения AP |
| `AP_REJECT_STATUS` | Стадия отклонения AP |
| `LA_ADD_ALL_MANAGERS` | Включать всю цепочку иерархии (не только прямого начальника) |
| `LA_EXCLUDE_DIRECTOR` | Исключить директора из фазы 3 |
| `LA_FORCE_DIRECTOR_IF_MANAGER` | Добавить директора в конец, если заявитель сам руководитель |

### Вкладка «Параметры отпуска»

| Ключ | По умолчанию | Описание |
|---|---|---|
| `LA_ENABLE_MIN_DAYS_CHECK` | `N` | Включить проверки дат |
| `LA_MIN_DAYS` | `14` | Минимальная длительность отпуска (дней) |
| `LA_MIN_DAYS_BEFORE_START` | `7` | За сколько дней до начала подать заявку |

### Логика `Approvers\ChangeStatus`

```
AP стадия изменилась?
        ↓
   → REJECT:
        требует REASON_FOR_REJECTION
        ER → ER_REJECT_STATUS
        стоп

   → APPROVE (PHASE = 1):
        хотя бы 1 HR одобрил среди всех AP фазы 1?
        → ДА: CreateApprovers.createPhase(2, erItem)
        → НЕТ: ждём

   → APPROVE (PHASE = 2):
        все AP фазы 2 одобрены?
        → ДА: CreateApprovers.createPhase(3, erItem, order=1)
        → НЕТ: ждём

   → APPROVE (PHASE = 3):
        есть следующий по ORDER?
        → ДА: CreateApprovers.createPhase(3, erItem, order+1)
        → НЕТ (последний): ER → ER_APPROVE_STATUS
```

### Логика `CreateApprovers`

```
createPhase(1):  берёт HR_MANAGERS → создаёт AP с PHASE=1
createPhase(2):  берёт PROJECT_MANAGERS → создаёт AP с PHASE=2
                 если пусто → сразу вызывает createPhase(3, order=1)
createPhase(3, order):
    есть UF_APPROVAL_PROFILE у заявителя?
    → ДА: находит ALP-ребёнка профиля с нужным ORDER → создаёт AP
    → НЕТ: берёт HEAD_OF_DEPARTMENT[order-1] → создаёт AP
           применяет настройки директора
```

## Тестовый стенд

Локальный стенд: **http://bitrix24.local**

### Смарт-процессы

| Роль | Название | entityTypeId |
|---|---|---|
| ER | Заявки сотрудников | 1046 |
| AP | Согласующие | 1050 |
| ALP | Профили согласования | 1054 |
| PP | Участники профиля | 1058 |

### Структура отделов

```
Руководство (ID=14)
└── Разработка
    ├── Иван Иванов   (заявитель)
    └── Пётр Петров   (руководитель отдела, defaultHeadRole)
HR (ID=12)
    └── Анна Сидорова (HR-менеджер)
```

### Сценарий полного согласования

1. `employee` создаёт заявку → `HEAD_OF_DEPARTMENT` и `HR_MANAGERS` заполняются автоматически
2. `employee` переводит ER в «В работе» → создаются AP фазы 1 (HR)
3. `hr` получает уведомление → переводит AP в APPROVE → создаются AP фазы 2 (РП, если есть) → создаётся AP фазы 3 (первый руководитель)
4. `head` получает уведомление → переводит AP в APPROVE → ER переходит в APPROVE
5. `employee` получает уведомление об одобрении → в инфоблоке `absence` появляется запись

### Сценарий отклонения

- Любой согласующий переводит AP в REJECT → обязательно заполняет `REASON_FOR_REJECTION` → ER сразу переходит в REJECT → `employee` получает уведомление
